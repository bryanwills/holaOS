---
title: Memory and Continuity
description: How holaOS separates memory into runtime continuity, durable recall, and workspace policy.
---

`holaOS` separates three concerns that are easy to blur together:

1. standing workspace policy
2. runtime continuity for resuming active work
3. durable memory for facts, procedures, preferences, identity, and recalled external context

That split is deliberate. It keeps runtime state, long-term memory, and workspace policy from getting mixed together.

## The current storage model

Today the runtime stores those concerns on different surfaces:

| Surface | Current shape | Purpose |
| --- | --- | --- |
| standing workspace policy | authored workspace files such as `AGENTS.md` | human-owned operating rules |
| runtime continuity | `turn_results`, request snapshots, persisted harness state, `.holaboss/memory/runtime/`, `.holaboss/memory/runtime/session-memory/` | cheap resume without replaying the full transcript |
| workspace durable memory | workspace-local markdown leaves plus semantic tree markdown under `.holaboss/memory/` | interaction-derived facts, procedures, blockers, and references |
| integration durable memory | global markdown leaves plus semantic tree markdown under `memory/` | recalled external context from Gmail, Notion, GitHub, Slack, and other integrations |
| secondary user-memory docs | `memory/preference/` plus optional `memory/identity/` | accepted long-lived user preferences and identity-style facts outside the main interaction or integration trees |
| canonical runtime profile | `state/control-plane.db` | profile state owned by the runtime, not by markdown memory |

Durable memory is now markdown-first and tree-based. The canonical content lives in markdown files. The runtime then derives semantic nodes, search docs, relations, and optional embeddings from that markdown so recall can work as hybrid RAG instead of flat file scanning.

Two path conventions exist at the same time:

- physical workspace-local files live under `<workspace-id>/.holaboss/memory/...`
- memory-tool namespaces refer to those same files as `workspace/<workspace-id>/...`

For example, a workspace interaction tree node can be addressed as `workspace/<workspace-id>/semantic/interaction/trees/<entity-slug>/content.md`, while the physical file lives under `<workspace-id>/.holaboss/memory/semantic/interaction/trees/<entity-slug>/content.md`.

That split between namespace and physical location matters:

- memory tools and APIs speak in memory-root-relative paths such as `workspace/<workspace-id>/...`
- the local filesystem stores workspace-local memory under `.holaboss/memory/`
- global memory such as integrations and preferences lives under the configured `memory/` root

Accepted user-memory proposals are still a separate write path. They currently promote durable preference memory into `memory/preference/` and may also update runtime-owned profile state in `state/control-plane.db`. `memory/identity/` remains a supported durable namespace, but it is an optional secondary surface, not the main current auto-write path. The background evolve path does not silently create user preference or profile memory on its own.

## Source-of-truth boundary

The source-of-truth split is deliberate:

- runtime execution truth lives in `<workspace-id>/.holaboss/state/runtime.db`
- canonical operator profile data lives in `state/control-plane.db`
- durable recalled memory lives in markdown under workspace-local `.holaboss/memory/` plus profile-global `memory/`
- semantic node rows, search docs, and embeddings are derived indexes over that markdown, not the canonical content
- standing workspace policy lives in `AGENTS.md`

That is what keeps continuity inspectable without turning workspace policy or durable memory into raw execution logs.

## How to think about the whole system

The most reliable mental model is:

1. `AGENTS.md` and authored workspace files define policy
2. runtime continuity keeps the current session resumable
3. durable memory stores what should be recallable later
4. retrieval reads the durable memory forest through `memory_retrieve`
5. background writeback turns repeated interaction signal into new durable leaves

That means the system is neither “just a vector database” nor “just a set of markdown files.” The markdown is canonical, while retrieval and browsing are layered on top of it.

## Retrieval model

Current recall is no longer “open `MEMORY.md`, then inspect a few leaves.”

The runtime exposes durable recall through `memory_retrieve`, which:

- plans whether the query should search interaction memory, integration memory, or both
- runs a vector first pass when recall embeddings are configured
- adds lexical support retrieval inside the best trees
- merges and reranks candidates into a bounded evidence pack for prompt-time RAG

Secondary `MEMORY.md` index files still exist as human-readable index surfaces and compatibility manifests, but they are not the primary durable memory shape anymore. The primary shape is the markdown forest itself.

## Browse vs recall

The runtime exposes memory through two different surfaces:

- the memory browser
  - a browseable and graphable projection of the semantic forest
  - shows `interaction/trees/...` and `integration/trees/...`
- `memory_retrieve`
  - a reasoning-oriented retrieval tool for query-time recall
  - returns bounded evidence instead of a full tree dump

Those surfaces complement each other. The browser is for inspection. `memory_retrieve` is for task-time context resolution.
