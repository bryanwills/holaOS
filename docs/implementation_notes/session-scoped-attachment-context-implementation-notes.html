Session-Scoped Attachment Context Implementation Notes

Design decisions

  • Implemented the first pass as append-only attachment provenance on session_messages.metadata.attachments rather than adding a separate active-attachment ledger.
  • Kept each attachment turn as its own timeline event, so reattaching the same file on a later turn produces a second preserved attachment record instead of collapsing the earlier one.
  • Projected prior attachment turns into runtime prompt context as a synthetic session-attachment timeline block, instead of reusing earlier files as if they were current-turn attachments.

Deviations

  • Did not implement explicit remove or replace semantics in this pass. Ongoing session attachment continuity is modeled only through append-only turn history.
  • Reused the existing prompt-context pipeline for later-turn resolution, rather than introducing a new harness payload field for persisted attachment artifacts.

Tradeoffs

  • This keeps timeline provenance aligned with text history and avoids the incorrect “carry forward as current attachments” behavior, but it also means older attachments are summarized into prompt context rather than inlined every turn.
  • Stored metadata is now the preferred source for history and future-run context, with a fallback to legacy agent_session_inputs.payload.attachments lookups so pre-migration sessions still work.
  • The runtime limits how many prior attachment turns and files are projected into prompt context to keep prompt growth bounded; truncated older turns remain in session history.

Open questions

  • Whether a later phase should expose explicit detach or replace operations, or whether append-only conversational artifacts are sufficient for the current product model.
  • Whether future harness work should allow selective re-inline of earlier attachment contents when the agent can prove it needs exact file text instead of just the timeline reference and staged path.
