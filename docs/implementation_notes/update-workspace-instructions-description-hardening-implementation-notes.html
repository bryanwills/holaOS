Update Workspace Instructions Description Hardening Implementation Notes

Design decisions

  • Hardened the tool by tightening text guidance first instead of changing the parameter schema shape.
  • Repeated the exact op names in the top-level tool description, the op field description, and the prompt guidelines so the model sees the same constraint through multiple channels.

Deviations

  • No schema-shape change yet. The implementation intentionally keeps the existing object-plus-union schema to avoid introducing provider-specific behavior differences before the description fix is evaluated.

Tradeoffs

  • Description hardening is lower risk across providers than moving immediately to a more complex discriminated schema.
  • This improves model guidance but does not add backend compatibility aliases, so a model can still fail if it ignores the guidance entirely.

Open questions

  • Whether to add a server-side alias from read to read_current if model-side adherence is still not reliable enough.
