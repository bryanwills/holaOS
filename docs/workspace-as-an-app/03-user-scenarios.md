# 真实场景：`Workspace As an APP`

文档原版缺少具体到能想象出截图的使用场景。这是补充的三个。每个场景刻意覆盖不同象限。

---

## 场景一：林夏 —— 单人内容创作者，把"X 话题建议器"分享给朋友

### Persona

林夏，单人运营一个小众科技领域的 X 账号（~8k followers），全职做内容已经一年。她在 Holaboss workspace 里搭了一个 agent，每天早上给她生成 5 条"今天可以发的话题"。用了 3 周，agent 已经很懂：
- 她的内容口味
- 她关注的小众主题
- 她过去发过什么（避免重复）

### Trigger

朋友 Mia 在 Slack 说：

> "你这个 agent 给你建议的话题质量比我现在用的任何工具都准。能不能也帮我看看我的账号？"

林夏想了一下：
- 让 Mia 用她自己的 workspace？看得见她的草稿、运营记录、其他 app —— ❌
- 让 Mia 复制一份 workspace？Mia 不用 Holaboss，配置半天才能开始 —— ❌
- 但她想给 Mia 一个"**只能用她这个工具**"的入口 —— ✅

### Flow

1. 林夏在 workspace 里找到"话题建议器" app，点右上角"对外公开"
2. 弹出小面板：
   - URL slug：`daily-takes`（她改了一下）
   - Auth：选"共享链接（需要 token）"
   - Publish：点了
3. 系统给她一个链接：`daily-takes.holaboss.app?key=mia-2024`
4. 复制到 Slack 发给 Mia
5. Mia 打开看到：
   - 干净的页面，标题"Daily Takes - by 林夏"
   - 左边输入"你的账号 handle + 想发的主题方向"
   - 右边出 5 条建议 + 每条配"用这个写"按钮
6. Mia **看不到**：林夏的草稿、林夏自己的话题记录、林夏其他 app、Holaboss 的任何运营 UI

### 价值瞬间

第二天 Mia 发了一条按建议写的 tweet，反响比平时好，回 Slack："**卧槽你这个工具好准**"。

林夏从来没想过自己能做"产品"。但现在她有一个**真的有人在用**的小工具。

### 暴露的产品要求

| 问题 | v1 必须有 |
|---|---|
| Mia 滥用怎么办？ | 共享链接必须**单条独立吊销** |
| Mia 一天调 200 次怎么算钱？ | 配额（即使粗暴：每天 N 次） |
| 想发给另一个朋友？ | 一个 app 能生成**多个**独立的共享链接 |
| URL 想自己改？ | slug 必须可改，且**改名后旧链接有友好下线页** |
| 朋友滥用了想看是谁干的？ | token 必须有 label / 标签，能看到用量 |

## 场景二：阿哲 —— 4 人 marketing 工作室，把"客户简报收集器"做成客户门户

### Persona

阿哲，4 人 B2B marketing 工作室创始人，专做 SaaS 公司的 LinkedIn 内容。workspace 里有：
- LinkedIn 集成
- 内容创作 app
- 客户档案 app（自己 vibe-coded 出来的）

他和 3 个同事**一起在这个 workspace 里干活**。

### Trigger

工作流痛点：每接一个新客户，要开 Zoom 让客户讲"行业 / 受众 / 内容方向 / 不想踩的雷 / 例子"。一次 ~1 小时，他记得不完整。

一个客户邮件随口问：

> "能不能给我个表单填？我自己能整理得更清楚。"

阿哲想到了一个比"表单"更好的东西 ——
- workspace 里已有一个 agent 能从对话抽信息存到客户档案
- 让客户**直接和这个 agent 对话**，比表单顺，比 Zoom 省时间
- 但客户绝对不能看到他**其他客户的档案**、运营记录、内容草稿

### Flow

1. 阿哲在 workspace 里 vibe-code 一个新小 app `client-intake`：和访客对话，把回答存到"待审核客户档案"
2. 点"对外公开"：
   - URL：`acme-marketing.holaboss.app/intake`
   - Auth：共享链接
   - Publish：点了
3. 每接一个新客户，**单独生成一个 token**（label = 客户名"Acme Corp"），邮件发给客户
4. 客户进去看到：chat 风格的简报收集器，agent 友好地问问题，~15 分钟
5. 阿哲在 workspace 里收到"待审核客户档案"，确认入库
6. **同事**在 workspace 里看到新客户，开始排内容
7. 同事**也想做一个新版** intake app，但**只有阿哲能点 Publish**

### 价值瞬间

两周后：
- 省下了 ~5 小时的 Zoom 时间
- 同事看到的客户资料**比阿哲 Zoom 完手记的还完整**
- 客户感觉"这家工作室很专业"

### 暴露的产品要求

| 问题 | v1 必须有 |
|---|---|
| 客户视角的品牌 | URL 必须能自定义 slug（不能 workspace UUID） |
| 多客户的链接管理 | 共享链接需要**管理面板**（list / label / 用量 / 吊销） |
| 数据归属 | 客户填进来的内容归属（v1：归 workspace） |
| 同事的权限 | Publish 权限边界 —— v1 owner only，必须显式 |
| 多 app 共存 | 一个 workspace 暴露**多个** app |
| 客户隔离 | 公众请求绝不能意外访问其他客户档案 —— B2B 硬底线 |

## 场景三：小川 —— 独立开发者，把"品牌声音分析器"发到 X 让陌生人玩

### Persona

小川，4 个月前离职，全职 indie hacker。X ~3k followers，目标是用小工具积累影响力。

workspace 里 vibe-coded 出一个 **Brand Voice Analyzer**：
- 你贴 5 条自己的 tweet
- 分析"品牌声音"：严肃/玩笑、技术/抒情、自信/谦虚
- 给雷达图 + 5 条改进建议

### Trigger

某个周五晚上他半开玩笑发 X：

> "我做了个小东西分析自己 tweet 的'品牌声音'，挺好玩的。AMA"

20 分钟内 12 个人回复："**能让我试试吗？**"

他需要：
- **匿名访问**（陌生人不会注册）
- **可分享的简单 URL**
- **abuse 保护**（否则免费 LLM 调用 = 一夜破产）

### Flow

1. 在 workspace 点"对外公开"
2. 设置：
   - URL：`voice.holaboss.app`
   - Auth：**匿名**
   - 日成本上限：**$20**（v1 必备，否则不敢开）
   - 单 IP 速率限制：每小时 10 次
3. 点 Publish
4. 发到 X：

   > "Try it free: voice.holaboss.app 🎯
   > Paste 5 tweets, get your brand voice profile."

5. 两小时后看"对外暴露"面板：
   - 访问量：420
   - 完成分析：187
   - 当前花费：$3.40
   - 一个 IP 试图调 200 次 → 已自动 throttle
6. X 上 6 个人发雷达图截图 + tag 他
7. 一个用户 DM："能不能做企业版？"
8. 周日他在 workspace 加新功能"导出 PDF 报告"，点 **"Publish update"** → 所有用户立即可用
9. 第二天被小 newsletter 收录，单日 1200 访问，他冷静把上限调到 $50

### 价值瞬间

第三天统计：
- X 涨 ~200 followers
- 收到 4 条"做企业版"DM
- 花费 $32（可控）
- 在 indie hacker 社区里被讨论

他对自己说：**这就是 holaboss 的正确用法**。做"产品"的路径从"想点子 → 写代码 → 部署 → 运营"压缩成"想点子 → 在 workspace 搭出来 → 一键 publish → 发链接"。

### 暴露的产品要求

| 问题 | v1 必须有 |
|---|---|
| 匿名 = 没 token 可吊销 | **强制速率限制 + 日成本上限**，v1 必备 |
| 一夜爆红怎么办？ | 成本上限可调，达到上限的行为可配置（拒绝/降级/暂停） |
| 滥用：有人贴脏话 | 必须有**一键 kill switch** |
| 想看谁在用？ | 基本 analytics 是 v1 一等公民 |
| 改了代码立刻生效？ | "Publish update" 语义必须明确 |
| SEO / 被搜到 | v1 默认 `noindex`，可由用户打开 |
| 上线时长？ | 匿名公开 = 长期挂，sandbox 不能 sleep 太久 |

## 三个场景合在一起的覆盖

| 维度 | 林夏 | 阿哲 | 小川 |
|---|---|---|---|
| **Audience** | 1 个朋友 | 几个企业客户 | 公开陌生人 |
| **Auth** | 共享链接 | 共享链接（多个） | 匿名 |
| **运营者数** | 1 人 | 4 人团队 | 1 人 |
| **风险类别** | 滥用 1 个 token | 客户隔离泄露 | 成本失控 / abuse |
| **价值类别** | 帮朋友 | 业务效率 | 增长 / 影响力 |
| **kill 紧迫度** | 几天内 | 几小时内 | 几分钟内 |

## 三个场景一起对文档的反馈

1. **"Publish" 是 publish app，"生成访问入口"是另一个动作。** 三个场景里这两件事都是分离的。文档把它们合在一起讲会出问题。

2. **不同 auth 模型有完全不同的 safety net 需求。**
   - 共享链接靠"吊销 token"
   - 匿名必须靠"速率 + 成本上限 + kill switch"

3. **analytics 是一等公民。** 三个 persona 都在场景里**主动想看**自己工具的使用情况。文档没把这列为 v1 验收项。

4. **vibe-coded 产物走 publish 是自然行为，不是边缘情况。** 阿哲和小川的 app 都是 vibe-coded 出来的。安全模型必须默认假设 publish 的代码是 vibe-coded。

5. **kill switch 必须是即时的。** 不是"撤销 publish 流程"，是"按一下立刻 down"。

6. **场景一是 C 端创作者，场景二是 B 端工作室，场景三是公开 indie tool。** v1 的安全模型按**最严格**的（B 端隔离）来做，C 端和 indie 自动受益。反过来不成立。

7. **三个场景都触发了"我能不能给两个不同的人不同的访问"** —— access 管理是产品里的**一等公民**，不是 publish 的附属配置。

8. **"URL slug 可自定义"** 在三个场景里都是用户主动想要的。slug 自定义 ≠ 自定义域名，是不一样的东西，v1 就该有。
