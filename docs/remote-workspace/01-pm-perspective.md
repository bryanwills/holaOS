# PM 视角：`Remote Workspace`

只看这一份。作为独立领域，不与 `Workspace As an APP` 合并叙事。

---

## 一、文档真正在卖什么"产品故事"

文档自己的措辞是"远程优先的工作区基础模型"。这是工程语言。翻成产品语言：

> **你的 workspace 不再是你电脑上的一个东西，而是一个一直都在的环境。**

潜台词：
- 关上电脑它也活着
- 在公司、在家、在手机上都是同一个
- agent 不会因为你下线而停下来
- "工作区"从一个**工具**变成一个**场所**

这是个**比文档表面看起来大得多的转变** —— 把这个产品从"AI 内容创作工具"重新定位成"一个一直在替你干活的环境"。这是一个**完全不同的产品类别**，对应**完全不同的留存模型和付费意愿**。

---

## 二、价值给谁、谁付钱

| 用户 | 角色 | 是否买单 | 重要度 |
|---|---|---|---|
| 内部 dogfood | 验证 | 不 | 跑通就行 |
| **现有客户（想要 hosted 连续性）** | **续费方** | **是** | **最重要** |
| **Admin / 团队所有者** | **企业付费方** | **是** | **金额最大** |
| agent 构建者 | 创作者 | 间接 | 决定深度使用 |
| 运营者 | 使用方 | 同上 | 决定日常体验 |

核心买单方是**两类**：
1. **续费的现有客户** —— 已经在用 desktop，"远程持续运行"是清晰的升级理由
2. **团队 / 企业账户** —— 这个产品**才让 team plan 真正成立**

**第二类是金额量级更大的故事**，但文档没把它拎出来讲。这是文档最弱的地方 —— 它把"团队"作为一种 user 列了，但没把"远程让 team plan 成立"作为**核心商业逻辑**写出来。

## 三、PM 必须先回答而文档列为 open 的事

文档列了 9 个 Key product decisions。压成一张立场表：

| # | 问题 | 我的立场 | 复杂度 |
|---|---|---|---|
| 1 | 运营者面 day-one | chat / outputs / integrations 完整；apps / files / automations 最小；不做 browser | Low |
| 2 | 集成管理模型 | **workspace-only 账号 + app reference**（不是 binding，不是 both） | Low |
| 3 | 记忆中心 vs app-local 边界 | 规则：**"删 app 该不该带走"** —— 是 → app-local，否 → 中心 | Low |
| 4 | 记忆读写所有权 | **single-writer per facet, all readers** | **Medium** |
| 5 | 更新模型（内部） | **live stateful** | Low |
| 6 | Runtime updater | **central + staged rollout + 通知不询问** | Low |
| 7 | 生命周期 | **sleep-wake，活动驱动** | **Medium** |
| 8 | Readiness / health 表达 | **两层 indicator + 详情页，不做 dashboard** | Low |
| 9 | 多操作者权限 | **owner + member 两层，无 surface 级粒度** | Low |

9 个里只有 2 个真有复杂度（#4 集成记忆架构、#7 sleep-wake 体验）。其他 7 个 PM 完全可以一周内拍板。

**Discussion 阶段 PM 应该带着这张表去，不要让会议变成"我们一起想想这 9 个问题怎么办"。**

## 四、第一刀切在哪（Wedge）

候选场景：

| Dogfood 场景 | 验证什么 |
|---|---|
| 内部 IM 团队把"运营机器人"做成远程 workspace | 团队场景 + 长期活着 |
| 一个 builder 把个人 workspace 远程化，跨设备使用 | "环境一直在" |
| **内部内容团队共享一个远程 workspace 做 social 运营** | **集中集成 + 集中记忆的价值** |
| 一个客户的"客服 agent"在远程 workspace 里 7×24 跑 | always-on / sleep-wake |

选**内部内容团队共享一个远程 workspace**作为第一刀。原因：
- 同时验证"团队场景"和"集中集成记忆"两个最值钱的承诺
- 内部用户 = 反馈最直接
- 内容场景是 holaboss 当下最强的产品形态
- 出问题不会影响外部客户

## 五、文档没写但必须想的事（沉默处即风险）

### 1. 计费模型完全缺席

远程沙箱有真实的持续成本（计算、存储、出入流量）。文档没提钱。

PM 必须答：
- 是订阅升级（desktop $X → remote $Y）？
- 是单独 SKU（远程是新产品）？
- 是按工作区数量？
- 是 always-on / sleep-wake 价差？
- 团队账户是按人头还是按工作区？

**这个产品的商业模式很可能是"按 active workspace 数量"**，因为这和成本最对齐。但 PM 必须先定。

### 2. Desktop 用户怎么办（cannibalization）

非目标里有一条"不做 promote local→remote"。真实用户**一定会问**：我的 desktop workspace 怎么变成远程的？

如果没有迁移路径，**desktop 用户会留在 desktop**，远程产品的增长被限制在 net new acquisition。

PM 必须答：desktop 是**被替代、并存、还是收敛**？我倾向**并存 + 远程作为默认推荐** —— desktop 留给"想要本地、想要隐私、想要更强 hardware"的人；远程作为新用户默认入口。

### 3. Onboarding 让用户选 remote / local 会伤新用户

文档说"用户新建 workspace 时选 remote"。

> 新用户在第一步面对一个"远程 vs 本地"的选择 = 大概率走错 / 弃流。

第一次接触产品的人**不知道自己应该选什么**。

正确形态：**默认远程**（除非用户主动装 desktop app）。"远程 vs 本地"不应该是 step 1 的分叉，应该是事后可切换的设置。

### 4. 工作区"挂了"的产品体验

远程工作区是个长期运行的远端进程，**它会挂**。PM 必须答：
- 工作区挂了，agent 在做的事丢不丢？
- 用户看到什么？
- 自动恢复吗？多久？
- 数据持久化保证是什么级别？

**承诺工作区一直活着 —— 这个承诺破掉一次，信任就没了。**

### 5. "在远程 workspace 里继续 vibe-code 新 App"的真实使用率

文档非常坚持这件事不能丢。但应该区分：
- **承诺级别**：必须保留（决定产品身份）
- **优化级别**：未必高（v1 不需要为这个能力做特别优化）

文档把它写得像高频功能，但实际可能 20% 的用户用，80% 的人只是消费 agent 输出。**v1 不要为这个能力做过度投入**。

## 六、衡量成功的指标

文档的 acceptance criteria 是工程口径。PM 视角的指标：

| 维度 | 指标 |
|---|---|
| **采用** | 新用户中选择远程的占比；现有 desktop 用户中尝试远程的占比 |
| **留存** | 远程 workspace 30/60/90 天**仍然 active** 的占比 |
| **使用深度** | agent 主会话日活、跨设备访问的用户占比 |
| **集中集成的价值验证** | 用户连接 2+ 集成的占比 |
| **集中记忆的价值验证** | 跨 app 引用集成记忆的 agent run 数量 |
| **信任** | 工作区 uptime、主动 churn 远程回 desktop 的比例 |
| **团队场景** | 一个 workspace 被 2+ 用户访问的占比（验证 team plan 假设） |

最后两条 PM 必须自己持有 —— 决定**这个产品未来 12 个月的故事**。

## 七、对文档的核心 push back

1. **缺少"为什么 remote"的产品价值主张** —— 文档说"远程优先"是个新模型，没写**对用户而言**它解决什么 desktop 解决不了的问题
2. **三个集中化的价值排序应该明确** —— "集成记忆集中"是最大差异化，应被拎出来作为产品核心承诺
3. **Team / Admin 场景的商业价值要写出来** —— 远程不是 desktop 的升级，是**让 team plan 成立的前提**
4. **Lifecycle 模型不能列为 open question** —— v1 必须是 sleep-wake，且要把唤醒做成产品时刻
5. **计费模型缺席必须补** —— 即使 v1 不变价，也要写清未来怎么算钱
6. **Onboarding 不应该让用户选 remote / local** —— 默认远程，本地是事后选择
7. **Desktop 用户的命运必须有立场** —— 并存 / 替代 / 收敛，三选一
8. **"挂了怎么办"的产品体验必须有定义** —— 这是"长期活着"承诺的最脆弱点
9. **vibe-coding 能力是产品身份，但未必是 v1 优化重点** —— 避免工程投入失衡
