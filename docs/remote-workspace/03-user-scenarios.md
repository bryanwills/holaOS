# 真实场景：`Remote Workspace`

文档原版缺少具体到能想象出截图的使用场景。这是补充的三个，每个故意打不同的角度。

---

## 场景一：陈璐 —— 数字游民内容创作者，工作区在所有设备上"一直都在"

### Persona

陈璐，28，数字游民。做小红书 + 公众号双平台内容（生活方式 / 旅行）。收入靠品牌合作 + 自营产品。每两周换一个城市。

工作设备：
- **MacBook Air** —— 主力创作
- **iPad** —— 灵感记录
- **iPhone** —— 随时改稿、刷 inbox

她之前用过 Holaboss 的 desktop 版，**但只能在 MacBook 上用**。

### Trigger

某次在巴厘岛，一个品牌方 DM 来问能不能下周交付一个 reel 脚本。她想立刻让 agent 开始构思，**但电脑没带**。手机回了句"明天给您"，凑合用了大半夜写大纲。

回到 MacBook 上想接着干，发现 agent 上下文都没了 —— 之前桌面版关了就 stop，今晚要重新 prompt 一遍背景。

心里那句："**这玩意儿为什么必须打开电脑才能用？**"

### Flow

1. 公司 announce remote workspace，她迁过去
2. 一周后又出门，在巴厘岛咖啡店打开 **iPad**
3. 浏览器登录 Holaboss → workspace 显示 "**Waking up... (~3s)**"
4. 进去：主会话**还在昨晚的对话**，agent 记得"明天交付的是关于日落咖啡馆的 reel 脚本"
5. 她在 iPad 上让 agent 直接写 3 个开篇 + 1 个完整脚本，10 分钟搞定
6. 回酒店打开 MacBook —— **同一个工作区状态，没有"重新同步"环节**
7. 睡前让 agent "夜里整理一下 inbox 里这周的品牌合作来信"
8. 关灯睡觉
9. 早上手机一打开看到 Holaboss 已经把 11 封 brand collab 邮件按重要性分好类、每封都有草稿建议

### 价值瞬间

她意识到 Holaboss **不再是"打开电脑才能用的工具"，而是"一直在替我干活的环境"**。

她跟朋友介绍这个产品时不再说"我用一个 AI 工具帮我写文案"，而是说："**我有一个 24 小时在线的内容助理。**"

### 暴露的产品要求

| 问题 | v1 必须有 |
|---|---|
| 第一次打开 3 秒 wake-up | 必须做成**有性格的产品时刻**（文案 + 微动画），不是 loading spinner |
| 跨设备切换状态以哪边为准？ | 需要明确的 "live state" 概念，冲突要可见可处理 |
| Agent overnight 跑的成本如何控制？ | per-workspace 成本观测 + 上限 |
| Agent overnight 决策的依据是什么？ | "agent 为什么把这封邮件标为 hot lead"必须可点开看 |
| iPad / 手机上的产品形态 | v1 至少要有可用的 web 形态，不能只是"桌面体验的缩水" |

## 场景二：沐光工作室 —— 4 人小品牌团队，"一个共享的工作环境"

### Persona

沐光，做手作首饰的 DTC 品牌。
- **江月** —— 创始人，owner
- **Anna / Bell / Cici** —— 内容 team members

主要平台：小红书 / 抖音 / 自营独立站。每周需要发 ~20 条内容。

之前的状态：
- 每人**各自**有 desktop 版的 Holaboss
- 每人**各自**连过的集成账号互不通（4 人共享一个小红书账号，每次发内容要"麻烦上一个连账号的人"）
- agent 跟 Anna 聊过的客户偏好 / 创意方向，Bell 完全不知道，要重新 prompt
- 内容 calendar 在 Notion 里另一份单独维护

### Trigger

某周一例会，Anna 和 Bell 同时端出"这周想发的小红书选题"，结果发现**有两条主题撞了**。Anna 已经写完，Bell 写了一半。

江月当场说了句："我们能不能**就一个工作区**，谁都能进去，agent 知道大家都在做什么？"

### Flow

1. 江月开了一个 team workspace，自己是 owner，把 3 人加为 member
2. **Workspace settings 里集成账号集中**：
   - 一个小红书账号
   - 一个抖音账号
   - 一个独立站后台
   - 任何 member 都能用，不用切来切去
3. 江月在 main session 跟 agent 长聊："我们这季度内容方向是**清新极简**，避开过于女性化的标签，受众是 25–35 岁有审美追求的女性"
4. Anna 周一打开 workspace 直接说 "帮我出 5 条这周可以发的小红书选题" —— agent **已经知道前面那段品牌方向**
5. Bell 周二打开 → 自动看到 Anna 昨天发了什么 → 自然避开重复
6. Cici 周三发现独立站后台需要改一个配置，但弹窗提示 "**需要 owner 权限**"
7. 江月晚上从家里用 iPad 打开 dashboard：这周已经发 8 条，还差 12
8. 月末江月发现 agent memory 里**自动多了一个"沐光品牌声音"条目** —— 这是 agent 从这个月跟 4 个人的对话里**自动归纳出来的**

### 价值瞬间

江月发现以前每周浪费在"咱们这周发什么了"的同步会上的 30 分钟没了 —— 信息天然在 dashboard 里。agent 越来越懂他们品牌的口味，**这种"懂"在 4 个人之间是共享的**。

她跟朋友介绍：「Holaboss 是**我们团队的第五个员工**。」

### 暴露的产品要求

| 问题 | v1 必须有 |
|---|---|
| Cici 能不能看 Anna 跟 agent 聊过的内容？ | main session 共享 / 私密边界必须显式 |
| 共享集成账号的"使用记录"谁能看？ | 集成账号操作要有 audit trail |
| 两人同时让 agent 出选题，agent 怎么处理潜在重复？ | agent 需要意识到 workspace 是多人的 |
| 集成 memory 里 agent 自动总结的"品牌声音"，江月想编辑/拒绝 | memory 必须可见 + 可编辑 + 可 veto |
| owner vs member 的具体权限分界 | v1 必须有明确清单，不是模糊的"管理 vs 使用" |
| 一个 member 离开公司怎么办？ | 权限回收 + 不影响 workspace 内她产生的 memory |

## 场景三：启明 —— Solo founder，"睡觉时还在干活的工作区"

### Persona

启明，35，SaaS 创业者，公司 4 个人，早期阶段。他自己负责 BD / 市场 / 销售。时间永远不够。

最大的痛点：**世界 24 小时在转，他必须睡觉**。
- 邮件来自 5 个时区
- LinkedIn 每天有 cold outreach + 真意向客户
- Slack 客户提问随时来
- Calendar 一直在被请求会议

早上起来要花 **1.5 小时**做"分类 + 回复 + 安排"。

之前在 desktop 版用 agent 写邮件，但 **desktop 关了 agent 就停**。

### Trigger

某个周日晚上正在和家人吃饭，手机推送一个客户的紧急 DM：

> "请问明天能不能给我们 demo？我老板要看"

他要赶紧从家庭聚会抽身回邮件、约 demo、协调日程。**这件事在他睡觉时会发生，在他陪小孩时会发生，在他健身时会发生**。

他读到 Remote Workspace 介绍里"long-running agent environment"那段时，整个人坐直。

### Flow

1. 创建 remote workspace，把 **Gmail / LinkedIn / Slack / Calendar** 一次性连上
2. 在 main session 配置 agent 的"夜班职责"：
   - 每小时检查新邮件，分类：Hot lead / Customer / Spam / Other
   - Hot lead 备好草稿，**但不发**
   - LinkedIn cold outreach 用他的口吻**自动拒绝**
   - LinkedIn 真实意向**只标记，等他看**
   - Slack 客户问题超 2 小时未回 → 推送 Telegram 通知
3. 关电脑睡觉
4. 早上 7:30 打开 holaboss web，**一屏总结**：

   ```
   过去 8 小时:
   📧 23 封邮件
       └─ 4 个 hot lead（草稿已备）
       └─ 1 个紧急客户问题（已 escalate 到你 Telegram）
       └─ 17 个 spam 已分类
   💼 LinkedIn: 11 个 DM
       └─ 2 个真意向（需要你确认）
       └─ 9 个 cold outreach 已自动回拒
   💬 Slack: 3 个客户问题，agent 已回复 + 标记
   ```

5. 花 **15 分钟**看 4 个 hot lead 草稿，微调后发出
6. 一周后他发现 agent 的判断**越来越准** —— integration memory 里它学到：
   - "启明对 enterprise lead 比 SMB 重视"
   - "启明回复风格简短直接"
   - "提到 Acme 客户的邮件优先级永远更高"

### 价值瞬间

启明从每天早上 **1.5 小时降到 30 分钟**。更重要的是：

**他不再担心"我睡着的时候错过了什么"**。

他第一次能在周末完全不看手机。

### 暴露的产品要求

| 问题 | v1 必须有 |
|---|---|
| Agent overnight 自动判断的失误怎么发现 + 反馈？ | agent 决策必须 **post-hoc 可追溯 + 可纠正** |
| Integration memory 学到的偏好想看 + 编辑 | memory 必须**对用户透明**，不能是黑箱 |
| 半夜 agent 卡死，4 小时邮件没处理 | 必须有 **liveness 心跳 + 失败通知** |
| 24h always-on 的 token 花费如何 cap | 必须有日 / 月成本上限 + 即将到达的预警 |
| 推送 Telegram 这种 out-of-workspace 通知 | 需要**通知路由层**，工作区能把信号送到工作区之外 |
| Agent 自动拒绝 cold outreach 万一拒错了 | 自动行为必须可**事后审查** |

## 三个场景的覆盖

| 维度 | 陈璐 | 沐光 | 启明 |
|---|---|---|---|
| **用户类型** | 个人创作者 | 4 人团队 | Solo founder |
| **核心需求** | 跨设备连续性 | 共享工作环境 | 自主长时间运行 |
| **Agent 角色** | 创作助手 | 团队第 5 个员工 | 24h 个人助理 |
| **驱动场景** | 移动办公 | 协作避撞 | 时区压迫 |
| **使用频次** | 一天数次 | 全员每天 | 持续在线 |
| **离开 desktop 的程度** | "电脑不在身边" | "不止一个人用" | "我睡觉它在干活" |

---

## 三个场景对文档的反馈

1. **"远程"在三个场景里不是"在云上"，而是**：
   - "不需要打开电脑"（陈璐）
   - "我们都能进去"（沐光）
   - "我睡着它还在干"（启明）
   
   文档应该用**用户能感知的语言**重写价值主张，而不是 "runtime contract / hosted environment" 这种工程语言。

2. **Sleep-wake 的"唤醒"是一个产品时刻，不是 loading state。** 陈璐场景里的 3 秒是产品的第一印象 —— 不设计 = 用户以为网卡了。

3. **"集成记忆集中"的真正价值在场景三显现：agent 学习用户的偏好。** 文档只把这个当架构决策，但场景里它是**最强的用户感知差异化**。这一点必须从"产品决策清单"升到"产品核心承诺"。

4. **Agent 自动行为的可追溯 / 可纠正性是三个场景共同的盲点。** 文档完全没碰，但每个 persona 都会问"agent 这么判断的依据是什么"。这是个 v1 必须解决的产品问题，不是后续优化。

5. **通知路由（out-of-workspace 通知）在启明场景是硬需求。** 远程工作区如果不能"伸出来找用户"，"24h 在线"的承诺就破了一半。

6. **多人 workspace 的 memory 边界（场景二）和单人 workspace 不一样。** 文档当前的"集中集成记忆"叙事假设了一个 owner —— 团队场景里"谁能看 / 编辑 / 拒绝 memory"是新问题。

7. **三个场景都隐含"工作区之外有东西"** —— 陈璐的设备切换、沐光的成员管理、启明的 Telegram 推送。文档把远程工作区当成一个内部完整的世界，但实际它**永远要和外面对话**。
