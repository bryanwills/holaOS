# 复盘结论

针对 `03-user-scenarios.md` 末尾遗留的 7 条反馈，收敛成 3 个落地结论。

---

## 结论 1：产品价值主张要重写成"三层独立"叙事

### 三个场景揭示的真正用户语言

| 场景 | 用户语言的"远程"是什么 | 维度 |
|---|---|---|
| 陈璐（数字游民） | "不需要打开电脑" | **设备无关** |
| 沐光（团队） | "我们都能进去" | **人无关** |
| 启明（solo founder） | "我睡着它还在干" | **时间无关** |

文档目前只表达了第一层（"hosted 替代 desktop"），后两层完全没讲出来。但**第二、三层才是这个产品真正想卖的东西** —— "团队第 5 个员工" / "24 小时在线助理"。

### 回写文档

把 Problem / Desired outcome 段改写，**用三层独立的语言**描述远程工作区的价值：

> 远程工作区让你的工作环境不再依赖：
> - **你坐在哪台设备前** —— 跨设备连续，状态在多端共享
> - **谁在使用它** —— 团队共用同一个工作环境与集成账号
> - **你是否清醒** —— agent 在你不在时继续运行，按你设定的规则处理事务

acceptance criteria 也要相应分别验收三层（不是只验"远程能跑"）。

## 结论 2：Agent 自动行为的可追溯 / 可纠正是 v1 必备

### 三个场景的实证

- **陈璐**：早上看到 agent 整理好的邮件优先级，想知道"凭什么排这个序"
- **沐光**：江月看到 agent 自动总结的"品牌声音"条目，想编辑 / 否决
- **启明**：agent 半夜自动回拒 cold outreach，他想看 "拒掉的是不是真的是 cold"

三个场景全都问：**agent 凭什么这么判断？**

文档当前完全没碰这件事。

### 为什么是 v1 必备而不是后续优化

远程工作区的核心承诺是 "agent 替你干活"。如果用户**看不到 agent 干了什么、为什么这么干**，整个产品的信任基础就建不起来：

- 用户不敢让 agent 自动执行 → 退化成"agent 建议，我手动确认"
- 退化后 = 远程工作区相对 desktop 没有本质优势
- **没有自动行为的可追溯性 = 远程工作区的产品身份失败**

### v1 必做

| 项 | 要求 |
|---|---|
| Agent 自动行为日志 | 每个自动决策必须可追溯到决策依据（哪条 memory / 哪个规则 / 哪个推理） |
| 决策点可编辑 / 可标错 | 用户能告诉 agent "你这个判断错了" |
| Memory 透明 + 可编辑 | agent 学到的偏好用户能看到、能改、能 veto |
| 决策回顾入口 | 单独一个 surface 看 "agent 过去 X 小时做了什么 + 为什么" |

### 回写文档

在 Desired outcome 加：

> 用户可以查看 agent 在工作区里产生的任何自动决策的依据，并对错误判断给出反馈以纠正未来行为。Agent 学到的偏好（integration memory）对用户透明、可编辑、可拒绝。

acceptance criteria 加：

> Agent 任一自动行为可被用户追溯到决策依据；用户可编辑或拒绝 agent 学到的任何 memory 条目。

## 结论 3：通知路由（out-of-workspace 通知）是 v1 必备

### 启明场景揭示的硬需求

启明的 24h 助理只有在**能找到他**的时候才有意义：
- Slack 客户问题超 2h → 推送 Telegram
- 半夜来了真的紧急邮件 → 推送
- agent 半夜卡死了 → 推送

**远程工作区如果不能"伸出来找用户"，"24h 在线"承诺破一半。**

陈璐场景里也有同样的痛点（agent 整理好的"重要邮件"她希望能在手机弹通知，不需要主动打开 web）。

### 文档完全没碰

文档把远程工作区当成一个**内部完整的世界**，但实际它**永远需要和外面对话**。

### v1 必做

| 项 | 要求 |
|---|---|
| 通知触发能力 | agent / cronjob / 集成事件能产生通知 |
| 通知路由配置 | 用户配置"什么级别的事件走哪个渠道" |
| 通知渠道（v1 至少 2 个） | 邮件 + Web Push 是底线；Telegram / Slack 是加分 |
| 通知可静音 / 可暂停 | 用户出门 / 度假可以暂停（避免轰炸） |

### 回写文档

Desired outcome 加：

> 远程工作区能在用户不在工作区时主动通知用户，通知渠道与级别可由用户配置。

acceptance criteria 加：

> 用户可配置至少两种 out-of-workspace 通知渠道，并定义触发通知的事件类型与级别。

## 还未收敛但已可见的事

这三条结论之外，场景里还揭示了几件**已经看到但本次不进 v1 范围**的事：

- **多人 workspace 的 memory 治理**（沐光场景）：member 如何看 owner 的 main session？member 离职怎么回收？这是个完整的 sub-area，建议作为 v2 单独立项。
- **跨设备状态冲突解决**（陈璐场景）：iPad 改了草稿、MacBook 也改了同一个草稿 —— 谁覆盖谁？v1 可以接受 "last write wins"，但 v2 需要明确策略。
- **vibe-coding 能力在三个场景里都没出现** —— 印证了 PM perspective 里的判断："vibe-coding 是产品身份的一部分，但不是高频使用功能"。v1 不为它做过度优化。
